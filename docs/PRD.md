# Rebalancer 产品需求文档（终稿）

## 1. 文档目标
定义一个功能克制、信息清晰、可直接落地开发的再平衡观测产品。

产品只聚焦三件事：
1. 基金配置录入
2. 再平衡观测
3. 多配置切换

---

## 2. 产品形态与定位
### 2.1 产品形态
1. 产品形态为 Chrome 插件。
2. 主交互容器为浏览器浮窗（Popup）。
3. 核心流程须在浮窗内闭环完成，不依赖外部页面跳转。

### 2.2 产品定位
1. 面向个人投资者的轻量再平衡观测工具。
2. 用最少输入获得可执行的调仓观察结果。
3. 不做交易执行，不做复杂分析，不做策略自动化。

---

## 3. 设计原则
1. 功能克制：所有功能直接服务“再平衡观测”。
2. 简单明确：字段命名直白，交互路径最短。
3. 结果可解释：每个指标都可追溯到输入与计算口径。
4. 视觉高完成度：简约、美观、克制，避免视觉噪音。
5. 信息优先级明确：关键数据（偏离、建议调仓）优先呈现。

---

## 4. 用户与核心场景
### 4.1 目标用户
有基金持仓、需要定期判断组合偏离并手动调仓的用户。

### 4.2 核心场景
1. 新建配置并录入基金与目标比例。
2. 查看当前偏离与建议买入/卖出金额。
3. 在多个配置之间切换观察。

---

## 5. 功能范围
### 5.1 必须交付（MVP）
1. 基金配置录入
2. 再平衡观测
3. 多配置切换

### 5.2 不在本期范围
1. 交易下单与券商联动
2. 回测、归因、风险分析
3. 自动调仓执行
4. 社区、资讯、社交功能
5. 复杂提醒与消息系统

---

## 6. 详细需求
## 6.1 基金配置录入
### 6.1.1 输入项
1. 配置名称
2. 总金额（人民币）
3. 起始日期
4. 持仓列表
   - 基金项：基金代码、基金名称（自动补全）
   - 现金项：名称（默认“现金”）
5. 每个持仓目标比例（0-100）

### 6.1.2 规则
1. 同一配置内基金代码不可重复。
2. 保存配置前必须满足：
   - 配置名称非空
   - 总金额大于 0
   - 至少 1 个持仓项
   - 目标比例总和等于 100
3. 起始日期若为非交易日，使用该日期之前最近交易日净值。

### 6.1.3 导入基金列表
1. 支持两种导入方式：
   - 粘贴 JSON 文本导入
   - 选择 JSON 文件导入
2. MVP 兼容格式：
   - 自选基金助手导出结构
   - 代码提取路径：`fundListGroup[].funds[].code`
3. 导入容错：
   - 仅接受非空字符串基金代码
   - 自动去重
   - 无效项跳过，不中断整体导入
4. 扩展要求：
   - 导入解析层采用解析器注册机制
   - 支持后续新增其他 JSON 格式解析器
   - 新增格式时不修改既有解析器逻辑

## 6.2 再平衡观测
### 6.2.1 输出信息（按配置）
1. 组合总市值
2. 累计收益（金额、收益率）
3. 持仓级指标：
   - 目标占比
   - 当前占比
   - 偏离（百分点）
   - 建议调仓金额（需买入/需卖出）

### 6.2.2 计算口径
1. 初始份额 = 初始分配金额 / 起始净值
2. 当前市值 = 当前净值 × 初始份额
3. 当前占比 = 持仓当前市值 / 组合总市值
4. 偏离 = 当前占比 - 目标占比
5. 建议调仓金额 = 目标市值 - 当前市值

### 6.2.3 刷新与异常
1. 支持手动刷新当前净值。
2. 支持基础缓存（建议 1 分钟粒度）减少重复请求。
3. 单只基金请求失败不影响其他数据展示。
4. 失败项须给出可读错误并支持重试。

## 6.3 多配置切换
### 6.3.1 功能
1. 创建多个配置
2. 配置切换查看
3. 设置主配置（默认打开）
4. 删除配置

### 6.3.2 体验要求
1. 切换操作反馈即时，状态明确。
2. 当前选中配置可视化清晰。

---

## 7. 数据与存储
1. 数据存储在本地（浏览器侧）。
2. 存储内容：
   - 配置列表
   - 主配置 ID
   - 净值缓存
3. 存储要求：
   - 重启后可恢复
   - 数据结构支持版本化演进

---

## 8. 非功能要求
1. 性能：常规规模（<=20 持仓）下流畅渲染与操作。
2. 稳定性：单点接口异常不导致页面不可用。
3. 可维护性：视图层与计算层解耦。
4. 视觉质量：
   - 整体简约、美观、克制
   - 排版统一、层级清晰
   - 色彩语义明确（正/负/中性）
   - 关键结论信息可一眼识别

---

## 9. 验收标准（MVP）
1. 用户可在浮窗内完成新建配置并成功保存。
2. 目标比例总和不为 100 时禁止保存，并明确提示。
3. 每个持仓可显示目标占比、当前占比、偏离、建议调仓金额。
4. 可创建至少 3 个配置并完成切换。
5. 可设置主配置并在下次打开时默认展示。
6. 单只基金净值获取失败时，其他持仓仍正常展示。
7. 基金导入支持 JSON 文本与 JSON 文件两种入口。
8. 对 `fundListGroup[].funds[].code` 结构可正确提取并去重导入。
9. 导入模块可新增第二种 JSON 解析器，且不改动第一种解析器行为。
10. 产品整体满足“Chrome 插件浮窗 + 简约高审美”标准。

## 附录 A：可靠 API 数据源
### A.1 基金实时估值与基础信息
1. 用途：基金名称、单位净值、净值日期获取。
2. Endpoint：
   - `GET https://fundgz.1234567.com.cn/js/{fundCode}.js?rt={timestamp}`
3. 返回格式：
   - JSONP：`jsonpgz({...})`
4. 关键字段：
   - `fundcode` 基金代码
   - `name` 基金名称
   - `dwjz` 单位净值
   - `jzrq` 净值日期
   - `gsz` / `gszzl` / `gztime` 估值相关字段
5. 成功判定：
   - 可解析 `jsonpgz(...)`
   - `name` 与 `dwjz` 存在

### A.2 历史净值（起始日回溯）
1. 用途：起始日或最近交易日净值获取。
2. Endpoint：
   - `GET https://api.fund.eastmoney.com/f10/lsjz?fundCode={fundCode}&pageIndex={page}&pageSize={size}`
3. 请求头要求：
   - `Referer: https://fund.eastmoney.com/`
   - `User-Agent: Mozilla/5.0`
4. 关键字段：
   - `ErrCode` 业务状态码（0 为成功）
   - `Data.LSJZList[]` 历史净值列表
   - `LSJZList[].FSRQ` 日期
   - `LSJZList[].DWJZ` 单位净值
5. 成功判定：
   - `ErrCode = 0`
   - `Data.LSJZList` 为数组

### A.3 数据接入通用要求
1. 每次请求设置超时（8-12 秒）。
2. 数值字段统一做类型转换与异常保护。
3. 失败请求需记录日志并向用户反馈可读错误。
4. 接口异常不阻断整体观测页面可用性。

## 附录 B：导入解析器约定
### B.1 解析器接口
1. `formatName: string`
2. `canParse(raw: unknown): boolean`
3. `extractCodes(raw: unknown): string[]`

### B.2 主流程
1. 读取输入（文本或文件）并 `JSON.parse`。
2. 按注册顺序选择首个可解析的解析器。
3. 提取基金代码并执行去重与校验。
4. 返回导入结果与错误信息。

### B.3 错误提示规范
1. JSON 非法：提示“JSON 解析失败”。
2. 无匹配解析器：提示“暂不支持该 JSON 格式”。
3. 代码为空：提示“JSON 中未找到有效基金代码”。
