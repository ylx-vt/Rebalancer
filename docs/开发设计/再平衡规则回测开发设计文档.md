# 再平衡规则回测开发设计文档

本文面向 `再平衡规则回测` 阶段一开发评审，说明功能落地时的模块边界、数据结构、计算流程、界面集成、异常处理和测试验收方案。

对应需求文档：[再平衡规则回测需求文档](../需求文档/再平衡规则回测需求文档.md)

## 1. 开发目标

在不影响现有再平衡观测能力的前提下，为同一持仓配置提供多种再平衡规则的历史模拟与对比。

本期交付目标：

1. 选择一个已有配置作为回测对象。
2. 设置回测起止日期和临时初始金额。
3. 添加并运行多个再平衡规则。
4. 输出规则对比表、收益曲线、回撤曲线和调仓记录。
5. 实际回测区间使用所有基金可估值数据的最大交集，不做净值不完整区间的解释性计算。
6. 历史净值缺失或规则配置不完整时，阻止生成误导性结果。

## 2. 当前工程基础

现有工程已经具备以下能力：

1. `src/domain/types.ts` 定义持仓、配置、净值、观测结果等核心类型。
2. `src/domain/calculation.ts` 负责当前组合再平衡观测计算和配置校验。
3. `src/services/fundApi.ts` 支持实时净值、指定日期之前最近可用历史净值获取。
4. `src/services/fundClient.ts` 支持基金数据缓存、批量加载和单基金失败隔离。
5. `src/services/storage.ts` 使用版本化 `AppState` 持久化配置和缓存。
6. `src/ui/App.tsx` 承载 Chrome Popup 内完整交互。

阶段一回测需要新增历史净值序列能力和回测引擎，不应把回测逻辑塞进现有 `calculatePortfolioObservation`，避免当前观测和历史模拟互相污染。

## 3. 建议目录结构

```text
src/
  domain/
    backtest.ts              # 回测引擎、指标计算、规则执行
    backtestTypes.ts         # 回测输入、规则、结果、错误类型
    calculation.ts           # 保持当前观测计算
    types.ts                 # 共享业务类型
  services/
    historicalNavSeries.ts   # 历史净值序列获取、对齐、缓存 key
    fundApi.ts               # 增加按区间获取历史净值序列的底层接口
    fundClient.ts            # 保留当前观测 quote 加载
    storage.ts               # 扩展最近一次回测参数和序列缓存
  ui/
    App.tsx                  # 在配置内增加回测入口和页面状态
    BacktestPanel.tsx        # 回测参数、规则列表、运行入口
    BacktestResultView.tsx   # 指标表、曲线、调仓记录
    Chart.tsx                # SVG 曲线组件或轻量图表组件
```

拆分原则：

1. 回测计算保持纯函数，输入完整净值序列，输出完整结果。
2. 接口请求、缓存、日期对齐放在 service 层。
3. UI 只负责参数收集、运行状态和结果展示。
4. 当前观测逻辑不依赖回测模块。

## 4. 核心数据模型

新增文件 `src/domain/backtestTypes.ts`。

```ts
export type BacktestRuleType =
  | "buy-and-hold"
  | "fixed-frequency"
  | "threshold"

export type BacktestFrequency = "monthly" | "quarterly" | "semiannual" | "annual";

export interface BacktestRule {

  holdingId: string;

  maxDrawdown: number;

```

说明：

1. `buy-and-hold` 对应不再平衡。
2. `fixed-frequency` 只按频率检查并执行再平衡。
3. `threshold` 每个交易日检查阈值。
4. `frequency-threshold` 只在频率检查日判断阈值。
5. 现金持仓作为 `kind: "cash"`，净值序列恒定为 `1`。
6. `riskFreeRatePercent` 默认值为 `2`，用于夏普比率计算，用户可在高级参数中修改。

## 5. 历史净值序列获取

### 5.1 底层接口扩展

当前 `fetchHistoricalNavBeforeOrOn(code, date)` 只能取某个日期之前最近一条净值。回测需要区间内连续交易日净值序列，建议在 `src/services/fundApi.ts` 新增：

```ts
export const fetchHistoricalNavSeries = async (
  code: string,
  startDate: string,
  endDate: string
): Promise<NavPoint[]>
```

接口策略：

1. 优先复用东财 `lsjz` JSON 接口。
2. 保留 HTML 备用接口作为 fallback。
3. 分页拉取，直到覆盖 `startDate` 之前最近可用净值，或超过最大页数。
4. 返回按日期升序排列的净值点。
5. 若无法覆盖起始日前最近净值或结束日前最近净值，抛出可读错误。

### 5.2 实际回测区间

用户输入的是期望回测区间，系统实际计算时必须收敛到所有基金可估值数据的最大交集时间范围。

实际回测区间生成规则：

1. 对每只基金截取用户期望区间内的有效净值序列。
2. 找出所有基金都有有效估值覆盖的最晚起始日期。
3. 找出所有基金都有有效估值覆盖的最早结束日期。
4. `actualStartDate` 和 `actualEndDate` 使用上述最大交集边界。
5. 若最大交集区间少于 2 个共同可估值日，回测不可运行。
6. 结果页必须同时展示用户选择区间和实际回测区间。

时间轴规则：

1. 回测只在实际回测区间内计算。
2. 时间轴使用实际回测区间内所有基金共同可估值日期。
3. 若单只基金在中间日期存在短暂缺口，可使用该日期之前最近可用净值前向填充。
4. 不对最大交集区间之外的净值缺失做解释性计算，也不生成不完整区间结果。

这样可以保证所有规则都基于同一段完整可比较的历史区间，避免用户误读为原始输入区间已经完整回测。


缓存 key：

```text
{code}:{startDate}:{endDate}
```

缓存策略：

1. 回测区间完全命中缓存时直接使用。
2. 回测区间被缓存区间覆盖时，可从缓存裁剪后使用。
3. 当前阶段不做跨缓存片段合并，避免复杂度过高。
4. 用户点击强制刷新时跳过序列缓存并覆盖写入。

## 6. 回测引擎设计

新增 `src/domain/backtest.ts`，核心入口：

```ts
export const runBacktest = (
  config: PortfolioConfig,
  input: BacktestInput,
  series: HoldingNavSeries[]
): BacktestResult
```

### 6.1 输入校验

运行前校验：

1. 配置存在且 `config.id === input.configId`。
2. 配置至少包含 1 个持仓。
3. 目标比例合计等于 100。
4. 基金持仓有有效代码。
5. `input.startDate < input.endDate`。
6. `initialAmount > 0`。
7. 至少 1 条规则，且最多建议限制为 6 条。
8. 每条规则名称非空。
9. 固定频率类规则必须有 `frequency`。
10. 阈值类规则必须有 `thresholdMode` 和大于 0 的 `thresholdValue`。
11. 无风险收益率必须是有效数字，默认 `2%`。

校验失败时返回表单错误，不进入历史数据请求。

### 6.2 初始建仓

对每条规则独立初始化持仓状态：

```text
初始分配金额 = initialAmount * targetPercent / 100
初始份额 = 初始分配金额 / 起始日净值
现金份额 = 初始分配金额
```

每条规则拥有独立份额状态，不能共享对象引用。

### 6.3 每日估值

按统一时间轴逐日循环：

1. 读取当日每个持仓净值。
2. 计算持仓市值、组合总市值、当前占比、偏离和相对偏离。
3. 计算当日组合净值点。
4. 判断是否触发再平衡。
5. 若触发，生成调仓记录并更新份额。
6. 更新回撤、日收益率等时间序列字段。

每日估值在调仓前记录当天组合状态；调仓记录保存调仓前市值和调仓后份额影响。下一交易日的估值体现调仓后持仓结构。

### 6.4 再平衡触发

触发判断函数：

```ts
const shouldRebalance = (
  rule: BacktestRule,
  date: string,
  index: number,
  state: BacktestHoldingState[]
): { triggered: boolean; reason: string }
```

规则口径：

1. `buy-and-hold` 永不触发。
2. `fixed-frequency` 在频率检查日触发。
3. `threshold` 任一持仓达到阈值即触发。
4. `frequency-threshold` 在频率检查日且任一持仓达到阈值时触发。

频率检查日口径：

1. 月度：每个自然月最后一个可估值日。
2. 季度：每个自然季度最后一个可估值日。
3. 半年：6 月和 12 月最后一个可估值日。
4. 年度：每年最后一个可估值日。

阈值口径：

1. `absolute-pp`：`abs(currentPercent - targetPercent) >= thresholdValue`。
2. `relative-percent`：`abs((currentPercent - targetPercent) / targetPercent * 100) >= thresholdValue`。

### 6.5 再平衡执行

执行口径：

```text
目标市值 = 调仓前组合总市值 * targetPercent / 100
调仓金额 = 目标市值 - 当前市值
调仓后份额 = 目标市值 / 当日净值
调仓换手金额 = min(买入金额合计, 卖出金额合计)
```

现金持仓也按目标比例参与恢复；现金净值恒定为 1。

记录字段：

1. 调仓日期。
2. 规则名称。
3. 触发原因。
4. 调仓前组合市值。
5. 每个持仓调仓金额。
6. 本次换手金额。

## 7. 指标计算

指标在单条规则回测完成后统一计算。

### 7.1 收益类

```text
累计收益率 = 期末金额 / 初始金额 - 1
年化收益率 = (期末金额 / 初始金额) ^ (365 / 实际天数) - 1
```

实际天数使用起始净值日期到结束净值日期的自然日差，最小值为 1。

### 7.2 回撤

按时间序列逐点计算：

```text
历史最高净值 = max(历史最高净值, normalizedValue)
回撤 = normalizedValue / 历史最高净值 - 1
最大回撤 = min(所有回撤)
```

展示时使用百分比绝对含义，存储时保留负数，便于曲线绘制。

### 7.3 波动率

```text
日收益率 = 当日总市值 / 前一可估值日总市值 - 1
年化波动率 = 日收益率标准差 * sqrt(252)
```

首个交易日 `dailyReturn = 0`，不参与标准差计算。

### 7.4 风险调整指标

```text
夏普比率 = (年化收益率 - 无风险收益率) / 年化波动率
卡玛比率 = 年化收益率 / abs(最大回撤)
收益回撤比 = 累计收益率 / abs(最大回撤)
```

口径说明：

1. 无风险收益率来自 `input.riskFreeRatePercent`，默认 `2%`。
2. 年化波动率为 0 时，夏普比率返回 `null`，UI 展示为 `--`。
3. 最大回撤为 0 时，卡玛比率和收益回撤比返回 `null`，UI 展示为 `--`。
4. 上述指标只用于规则历史表现参考，不用于推荐最佳规则。

### 7.5 扩展统计

```text
日胜率 = 正收益交易日数量 / 有效交易日数量
最佳单日收益 = max(日收益率)
最差单日收益 = min(日收益率)
最大连续回撤天数 = 从回撤开始到净值创新高前的最长自然日数
年化换手率 = 调仓总金额 / 平均组合市值 / 实际天数 * 365
平均单次调仓金额 = 调仓总金额 / 再平衡次数
```

再平衡次数为 0 时，平均单次调仓金额返回 `0`。

### 7.6 调仓统计

```text
再平衡次数 = rebalanceRecords.length
调仓总金额 = sum(rebalanceRecords.turnoverAmount)
```

调仓总金额采用换手金额口径，避免买入卖出双边重复统计。

## 8. UI 集成设计

### 8.1 入口

规则回测不作为 Popup 一级入口，不与再平衡观测并列。入口放在具体配置内，作为当前配置的子功能。

建议入口位置：

1. 当前配置标题区的次级按钮：`规则回测`。
2. 配置卡片操作区：`查看观测`、`规则回测`、`更多`。
3. 配置更多菜单内保留入口，但不作为唯一入口。

用户从某个配置进入后，回测对象锁定为该配置。若需要回测其他配置，先切换当前配置或从对应配置进入。

### 8.2 回测参数区

参数区包含：

1. 当前配置摘要。
2. 期望开始日期。
3. 期望结束日期。
4. 初始金额。
5. 高级参数：无风险收益率。
6. 强制刷新历史数据开关。

初始金额默认取配置 `totalAmount`，用户修改后只影响本次回测。

无风险收益率默认 `2%`，用户修改后只影响本次回测。

### 8.3 规则编辑区

规则列表支持：

1. 添加规则。
2. 删除规则。
3. 修改规则名称。
4. 选择规则类型。
5. 选择频率。
6. 设置阈值模式与阈值数值。

默认给出 3 条规则：

1. 不再平衡。
2. 月度再平衡。
3. 偏离 5pp 触发。

规则数量建议限制为 1 到 6 条，避免 Popup 图表和表格过载。

### 8.4 结果展示

结果区分为三个 tab：

1. `指标`：规则对比表。
2. `曲线`：收益曲线和回撤曲线。
3. `调仓`：按规则查看调仓记录。

指标表字段：

1. 规则名称。
2. 期末金额。
3. 累计收益率。
4. 年化收益率。
5. 最大回撤。
6. 年化波动率。
7. 夏普比率。
8. 卡玛比率。
9. 再平衡次数。
10. 调仓总金额。

扩展指标字段：

1. 日胜率。
2. 最佳单日收益。
3. 最差单日收益。
4. 最大连续回撤天数。
5. 收益回撤比。
6. 年化换手率。
7. 平均单次调仓金额。

结果顶部必须展示：

1. 用户选择区间。
2. 实际回测区间。
3. 无风险收益率。
4. “回测仅代表历史模拟，不代表未来表现。”

曲线绘制：

1. 使用轻量 SVG 折线，不引入重型图表库。
2. 同一张图最多展示 6 条规则。
3. 收益曲线使用 `normalizedValue`。
4. 回撤曲线使用 `drawdown`。
5. 悬停展示日期和各规则数值。

调仓记录：

1. 支持按规则筛选。
2. 每条记录展示日期、触发原因、调仓前市值、换手金额。
3. 展开后展示每个持仓买入或卖出金额。

## 9. 状态流转

UI 状态建议：

```ts
type BacktestRunState =
  | { status: "idle" }
  | { status: "validating" }
  | { status: "loading-data" }
  | { status: "running" }
  | { status: "success"; result: BacktestResult }
  | { status: "error"; message: string };
```

运行流程：

1. 用户点击运行。
2. 校验配置、日期、金额和规则。
3. 保存最近一次回测参数。
4. 获取并缓存历史净值序列。
5. 计算所有基金可估值数据的最大交集区间。
6. 执行每条规则回测。
7. 展示结果。

错误时保留用户输入，允许修改后重试。

## 10. 异常处理

必须阻止运行的异常：

1. 没有可用配置。
2. 目标比例不等于 100。
3. 基金代码为空或重复。
4. 开始日期不早于结束日期。
5. 初始金额小于等于 0。
6. 规则配置不完整。
7. 所有基金无法形成有效最大交集区间。
8. 最大交集区间内共同可估值日期不足 2 个。

可读错误示例：

```text
所选基金在 2023-01-01 至 2025-12-31 内没有共同可回测区间。
```

```text
规则“季度 + 5pp”缺少阈值设置。
```

```text
实际回测区间内共同可估值日期不足 2 个，请调整日期范围。
```

接口失败处理：

1. 展示失败基金代码和接口错误。
2. 支持重试。
3. 不生成部分规则结果，避免用户误解为完整回测。

## 11. 存储与兼容

存储变更保持 `schemaVersion: 1`，通过 `normalizeState` 对新增字段做可选兼容。

新增字段：

1. `historicalSeriesCache`
2. `lastBacktestInput`

兼容规则：

1. 旧状态缺少新增字段时初始化为空。
2. 回测参数保存失败不影响当前结果展示。
3. 历史序列缓存可被清空，不影响配置数据。
4. 不修改已有 `fundCache` 和 `historicalCache` 含义。

## 12. 开发拆分

建议按以下顺序开发：

1. 新增 `backtestTypes.ts`，定义输入、规则和结果结构。
2. 新增 `backtest.ts`，完成纯计算引擎和单元测试。
3. 扩展 `fundApi.ts`，支持历史净值区间序列。
4. 新增 `historicalNavSeries.ts`，负责缓存、对齐和错误归一化。
5. 扩展 `storage.ts`，保存最近一次回测参数和序列缓存。
6. 拆出 `BacktestPanel.tsx` 和 `BacktestResultView.tsx`。
7. 在 `App.tsx` 的配置视图内增加入口、状态流转和结果挂载。
8. 补充样式和响应式处理。
9. 完成验收用例和构建验证。

## 13. 测试计划

### 13.1 计算层测试

覆盖：

1. 不再平衡时份额全程不变。
2. 固定频率月度、季度、半年、年度检查日识别正确。
3. 绝对百分点阈值触发正确。
4. 相对偏离阈值触发正确。
5. 固定频率 + 阈值只在检查日触发。
6. 调仓后各持仓占比回到目标比例。
7. 最大回撤、年化收益、年化波动率计算正确。
8. 现金持仓净值恒定为 1。
9. 夏普比率、卡玛比率、收益回撤比计算正确。
10. 年化波动率或最大回撤为 0 时，相关比率返回 `null`。

### 13.2 数据层测试

覆盖：

1. JSON 历史接口分页解析。
2. HTML 备用接口解析。
3. 非交易日起止日期向前寻找最近净值。
4. 缓存完整命中。
5. 缓存覆盖区间裁剪。
6. 某基金缺少起点净值时报错。
7. 多基金最大可估值交集区间计算正确。
8. 实际回测区间不足 2 个共同可估值日时报错。
9. 交集区间内时间轴对齐和前向填充。

### 13.3 UI 验收

覆盖：

1. 无配置时展示空状态。
2. 默认生成 3 条规则。
3. 用户从配置内进入规则回测。
4. 用户可添加至少 3 条规则并运行。
5. 加载历史数据时按钮展示 loading。
6. 结果顶部展示用户选择区间、实际回测区间和无风险收益率。
7. 结果表展示核心指标和扩展指标。
8. 收益曲线和回撤曲线能区分不同规则。
9. 调仓记录可追溯日期和金额。
10. 错误信息明确且支持重试。
11. 切回再平衡观测不受回测结果影响。

## 14. 风险与取舍

### 14.1 历史净值数据量

长区间、多基金、多规则会增加请求量和内存占用。本期限制：

1. 规则最多 6 条。
2. 单次回测基金数沿用配置持仓数，不做批量配置回测。
3. 历史序列缓存只按完整区间或覆盖区间复用。

### 14.2 Popup 展示空间

Popup 空间有限，结果页采用 tab 分区，不把指标表、曲线和调仓记录同时堆叠展示。

### 14.3 指标解释

回测结果必须展示提示：

```text
回测仅代表历史模拟，不代表未来表现。
```

该提示放在结果区固定位置，避免用户把规则对比理解为投资建议。

### 14.4 交易费用

本期不建模交易费用、申购费、赎回费和分红再投资复杂口径。结果文案中明确本期为“不含交易费用的历史模拟”。

## 15. 评审关注点

评审时需要确认以下决策：

1. 回测规则数量上限是否接受 6 条。
2. 阈值触发是否按每日共同可估值日检查。
3. 固定频率检查日是否采用自然周期最后一个可估值日。
4. 规则回测入口是否固定放在配置内，而不是 Popup 一级入口。
5. 实际回测区间是否采用所有基金可估值数据的最大交集。
6. 无风险收益率是否默认 `2%`，并放在高级参数中。
7. 图表是否采用自研轻量 SVG，不引入图表库。
8. 回测缓存是否只做区间覆盖复用，不做片段合并。
